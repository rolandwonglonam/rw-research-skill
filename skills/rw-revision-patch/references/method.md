# 方法

## 输入门

- 固定的 Markdown 基础文稿。
- 用户批准的修改范围。
- 审稿意见、问题 ID 或其他修改理由。

## 流程

1. `anchor` 创建新副本和 manifest，不覆盖基础文稿。
2. 根据批注和上下文确定完整论证单位。被标出的词句只定位问题；需要时把同段前后句纳入批准块的 replace operation。
3. 对解释类批注检查主体、对象、行动、发生环节、作用关系、适用边界和当前意义；不以增加字数作为完成标准。
4. 人或 Agent 只为批准块生成 replace operation。
5. `check` 一次验证全部 operation。
6. `apply` 在所有 precondition 通过后生成新文稿。
7. 人工复核修改块，再用 report 检查保留比例和批注是否真正关闭。

## 原子性

任何一个 operation 的 block ID、hash 或格式错误，整批 Patch 失败。脚本不应用已通过的那一部分。

## 结构修改

新增、删除、重排章节不能用第一版 Patch 完成。先向用户说明结构变化，再建立单独任务和验收范围。

## 批注范围与论证范围

- 批注范围说明问题被读者看到的位置。
- 论证范围包括理解和修复该问题所需的前提、命题、解释、边界和后果。
- Patch 范围仍由用户批准。论证范围超出已批准块时，不截断解释，也不静默扩大范围；先提出范围变更。
