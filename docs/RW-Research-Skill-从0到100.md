## 先看这里

RW Research Skill 通过 Agent Skills 文件运行，不需要网站或单独服务器。这里的“部署”，就是把 Skill 安装到 Codex 或其他支持 Agent Skills 的 Agent 中。

如果你的电脑已经能运行 `node` 和 `npx`，部署只需要下面两步。

第一步，在 Terminal 或 PowerShell 中安装4个公开入口：

```bash
npx -y skills add rolandwonglonam/rw-research-skill -g --all
```

第二步，新建一个 Agent 任务。在 Codex 中输入：

```text
$rw-research-router 新手入门
```

在其他支持 Agent Skills 的 Agent 中，可以直接说：

```text
请使用 rw-research-router 带我新手入门。
```

完成后，你不需要理解内部21个 Skill。v0.11.0 默认只展示4个公开入口：

- `rw-research-router`：研究启动与推进。
- `rw-paper-extractor`：论文精读与证据。
- `rw-research-referee`：研究设计与审查。
- `rw-phd-write`：论文写作与投稿。

如果安装成功但入口没有出现，先新建一次对话，再重新调用。

## 这份手册怎样使用

你可以按自己现在的位置开始，不需要从第一页读到最后一页。

- 还没安装：从“20—40：完成部署”开始。
- 已安装但不知道怎么说：从“40—60：完成第一次真实任务”开始。
- 已经在用，但经常断在不同对话里：从“60—80：推进一个研究项目”开始。
- 已经能完成任务，但不知道如何核验：从“80—100：建立证据和验收”开始。
- 现在遇到报错：直接看“常见问题”。

这里的“100”表示你能够安装、提交材料、推进一个任务、核验结果、保存状态，并知道什么时候必须停止。Skill 仍不能独立完成全部科研工作。

# 0—20：先知道自己在使用什么

## 1. RW Research Skill 是什么

RW Research Skill 是一组科研工作流说明。它告诉 Agent 在面对研究想法、论文、数据、方案、草稿或审稿意见时：

- 先判断当前瓶颈。
- 只选择一个主任务。
- 说明需要哪些材料。
- 区分事实、解释、推断和未知事项。
- 在证据不足时停止，不补造论文、数据、DOI 或期刊要求。
- 完成当前一步后，再给一个下一步。

安装不会增加新的大语言模型，也不会替你建立在线账户。你正在使用的 Agent 会读取这些工作规则并执行任务。

## 2. 它和普通对话有什么不同

普通对话容易从你的文件类型开始。例如，你上传了一篇论文，Agent 可能直接总结。

RW Research Skill 要求先看你的目的。如果你想核验某个主张，它应该定位原文和证据。如果你想检查研究设计，它应该先看研究问题、样本、测量和分析。如果你想修改论文，它应该先确认章节功能、来源和修改范围。

它先确认你当前最需要解决的问题，再处理文件本身。

## 3. 4个公开入口

| 入口 | 你什么时候使用 | 你需要提供什么 | 主要结果 |
| --- | --- | --- | --- |
| `rw-research-router` | 不知道从哪里开始，或者需要判断下一步 | 想法、已有材料、当前卡点或目标 | 阶段判断、一个主入口、所需材料、停止条件 |
| `rw-paper-extractor` | 需要精读 PDF、提取图表、建立论文卡片或核验主张 | PDF、补充材料、待核验问题 | 结构化提取、原文位置、缺失项、复核状态 |
| `rw-research-referee` | 需要检查综述、研究设计、数据、统计报告或结论 | 方案、协议、数据说明、统计结果或结论 | 按严重程度排序的问题、结论影响、修复动作 |
| `rw-phd-write` | 需要写作、局部修改、处理审稿意见或准备投稿 | 草稿、来源、研究问题、修改范围和目标要求 | 论证骨架或正文、来源缺口、修改说明、未解决项 |

普通用户只需要记住这4个入口。内部 Skill 由入口根据任务调用。

## 4. 它不能替你做什么

RW Research Skill 不能替代：

- 研究者选择问题和承担学术责任。
- 导师、伦理委员会、统计师或临床专业人员的审批。
- 对论文全文、数据和来源的合法访问。
- 对当前期刊政策、API、费用和报告规范的实时核验。
- 真实实验、数据收集和人工复核。
- 作者对最后一版文字、引用和结论的确认。

材料不足时，可以得出“暂时无法判断”。不要为了完整补答案。

## 5. 公开包和私人材料

公开仓库只保存通用研究方法、公开来源入口、包内规则和合成测试输入。它不包含个人论文、数据、导师反馈、研究项目状态或私人工作区。

你提交给 Agent 的文件只作为当前任务输入，安装本身不依赖这些文件。文件是否会发送给模型服务，取决于你使用的 Agent、模型提供方和账户设置。涉及未发表论文、受限数据或个人信息时，先确认所在机构和所用工具的要求。

不要把真实论文、未发表数据、个人路径、账号或凭据提交到公共 GitHub Issues 或 Pull Requests。

# 20—40：完成部署

## 6. “部署”在这里是什么意思

这套 Skill 没有需要启动的 Web 服务，也没有需要配置的 RW 后端。

部署包括5件事：

1. 准备一个支持 Agent Skills 的 Agent。
2. 确认电脑可以运行 `node` 和 `npx`。
3. 从公共仓库安装 Skill。
4. 新建对话，让 Agent 重新加载入口。
5. 运行一次新手测试。

## 7. 安装前准备

你需要：

- 一台可以使用命令行的电脑。
- 可以访问 GitHub 和 npm 的网络。
- 一个支持 Agent Skills 的 Agent，例如 Codex。
- Node.js 和随附的 `npx`。

### macOS

打开 Terminal。可以在 Spotlight 中搜索“Terminal”。

### Windows

打开 PowerShell。可以在开始菜单中搜索“PowerShell”。

### 检查 Node.js 和 npx

依次运行：

```bash
node --version
npx --version
```

如果两条命令都返回版本号，可以继续。

如果出现“command not found”“不是内部或外部命令”或类似提示，先从 Node.js 官方页面安装 LTS 版本：

[Node.js 下载页](https://nodejs.org/en/download)

安装后关闭并重新打开 Terminal 或 PowerShell，再运行上面的两条检查命令。

本项目没有在公开说明中声明最低 Node.js 版本。这份手册不另行指定一个未经项目测试的最低版本。

## 8. 查看可以安装的入口

先运行：

```bash
npx -y skills add rolandwonglonam/rw-research-skill --list
```

v0.11.0 的默认列表应显示4个公开入口。只看到4个属于正常情况。

如果命令显示的仓库名称与 `rolandwonglonam/rw-research-skill` 不一致，先停止，不要继续确认安装。

## 9. 选择安装方式

### 方式A：安装4个公开入口

大多数学生使用这一种：

```bash
npx -y skills add rolandwonglonam/rw-research-skill -g --all
```

`-g` 表示写入全局 Skill 目录。安装后可以在不同项目中调用。

### 方式B：只安装研究启动入口

如果你只想先试用：

```bash
npx -y skills add rolandwonglonam/rw-research-skill -g --skill rw-research-router
```

只安装一个入口时，它仍会完成当前证据允许的部分。需要其他入口时，它会给出下一步说明。

### 关于安装命令

`npx -y` 会下载并运行 `skills` CLI。运行前要确认仓库地址和安装范围。公共安装路径以 GitHub 仓库为准，不使用来源不明的压缩包。

普通用户不需要安装或显示17个内部 Skill，也不需要复制维护者的私人 Research Lab。

## 10. 让 Agent 重新加载

安装结束后：

1. 关闭当前对话。
2. 新建一个对话或任务。
3. 在 Codex 中输入：

```text
$rw-research-router 新手入门
```

如果客户端不使用 `$` 形式，输入：

```text
请使用 rw-research-router。先判断我现在处于哪个研究阶段，每次只问我一个问题。
```

## 11. 第一次部署测试

新手入口的第一轮不应该把21个 Skill 全部列给你，也不应该要求你自己挑选内部流程。

它应该先确认：

- 你现在手上有什么。
- 这次想得到什么结果。
- 当前最卡住的一件事。

你可以只回答：

```text
我只有一个研究想法，还没有文献，也不知道怎样形成研究问题。
```

部署测试通过的最低标准：

- Agent 能识别 `rw-research-router`。
- 它只推进一个当前任务。
- 它说明还需要什么材料。
- 它给出停止条件和完成标准。
- 它没有编造论文、数据、DOI 或研究结果。

如果 Agent 直接输出一份完整论文，这次测试没有通过。重新说明：

```text
不要直接写论文。请按 rw-research-router 判断当前阶段，只推进一个主任务。
```

## 12. 更新

检查已安装 Skill 是否有更新：

```bash
npx -y skills check
```

执行更新：

```bash
npx -y skills update
```

更新后新建一次对话。版本变化时，先看仓库 README 和验证记录，不把旧手册中的数量和功能当作当前事实。

## 13. 部署验收

完成下面的检查后，部署阶段结束：

- [ ] `node --version` 返回版本号。
- [ ] `npx --version` 返回版本号。
- [ ] 安装命令没有报错。
- [ ] 新建对话后能识别 `rw-research-router`。
- [ ] 默认只看到4个公开入口。
- [ ] 新手入口只问当前材料和目标。
- [ ] 你知道如何运行 `skills check` 和 `skills update`。

# 40—60：完成第一次真实任务

## 14. 不要从“帮我做完整篇论文”开始

第一次使用时，选一个可以在一次任务中检查的结果。

可以选择：

- 把一个兴趣改成可继续讨论的研究问题。
- 从2篇 PDF 中提取研究设计和主要结果。
- 检查一份研究方案中最影响结论的问题。
- 根据现有来源修改一个小节。
- 核验3条主张是否得到引用支持。

不要把选题、检索、精读、统计、写作和投稿同时交给一个入口。

## 15. 贯穿示例

下面使用一个合成示例，只演示操作。这里没有给出实际研究建议，也没有声称已经找到相关证据。

```text
研究兴趣：大学生睡眠与学业表现。
现有材料：一个想法，尚未整理文献。
当前目标：先得到一个可以继续检索和讨论的研究问题。
```

### 第一步：让 Router 判断当前任务

在 Codex 中输入：

```text
$rw-research-router
我想研究大学生睡眠与学业表现。目前只有这个想法，没有整理文献。
这次只需要形成一个可以继续检索和讨论的研究问题。
请告诉我还缺什么，每次只推进一步。
```

这一步应先处理对象、构念、边界和可证伪条件。它不应假装已经知道文献结果。

本步完成标准：

- 原始兴趣被保留。
- 对象和核心构念得到说明。
- 未确定的边界被列出。
- 形成一个暂定研究问题，或说明为什么还不能形成。
- 下一步只有一个。

### 第二步：根据研究问题处理论文

获得论文全文后，使用：

```text
$rw-paper-extractor
下面是与暂定研究问题有关的2篇 PDF。
请先建立适合这个用途的提取字段，提取研究设计、样本、睡眠测量、
学业结果、主要结果和限制。关键字段要给出页码或表图位置。
无法从全文确认的内容标记为“无法判断”，不要补全。
```

本步完成标准：

- 每篇论文有稳定标识。
- 提取字段与后续用途有关。
- 方法、结果和作者解释分开。
- 关键字段有原文位置。
- 未报告、零事件、不适用和无法判断没有混写。

### 第三步：检查研究设计

形成初步方案后，使用：

```text
$rw-research-referee
这是我的研究问题和初步方案。
请先提取我想做出的最强结论，再检查选择、混杂、测量、缺失、
分析和选择性报告问题。每个问题说明它会怎样影响结论，
并区分阻断、重大修改、一般修改和报告澄清。
```

本步完成标准：

- 审查具体结论，语言风格不在本轮范围内。
- 每条批评说明结论影响。
- 至少列出2个替代解释。
- 修复动作和无法修复部分分开。
- 关键信息不足时，不给出“通过”判断。

### 第四步：根据现有证据写作

已有研究问题、来源和段落任务后，使用：

```text
$rw-phd-write
这是我的研究问题、来源和现有小节。
本次只修改这个小节，目标是说明现有研究怎样测量睡眠和学业表现，
以及这些测量差异为什么限制比较。
不要增加我没有提供或没有核验的引用。
请先给出论证骨架，再写正文，并列出仍缺的证据。
```

本步完成标准：

- 先说明小节承担的功能。
- 事实性主张能回到来源。
- 结果、作者解释和当前推断分开。
- 没有把证据缺口写成通用意义句。
- 修改没有越过约定范围。

## 16. 怎样向 Agent 提供材料

优先提供3类信息：

1. 你现在要得到的结果。
2. 你已经有的材料。
3. 不能越过的范围。

一个可复制的输入模板：

```text
当前任务：

我已经有：

这次需要的结果：

允许使用的来源：

不要做：

截止位置：
```

“截止位置”表示这次做到哪里停止。例如：

- 只形成检索概念，不运行数据库检索。
- 只提取2篇论文，不扩展到整批文献。
- 只审研究设计，不改文字。
- 只修改一个小节，不改其他章节。

## 17. 怎样判断第一次任务完成

第一次任务是否完成，看结果能否继续使用。文字长短不作判断依据。

检查：

- [ ] Agent 保留了你的原问题。
- [ ] 输出只处理当前任务。
- [ ] 事实、解释、推断和未知事项分开。
- [ ] 关键结论有来源位置，或明确标记待核验。
- [ ] 材料不足时没有补造答案。
- [ ] 下一步只有一个。

# 60—80：推进一个研究项目

## 18. 研究流程按项目调整

RW Research Skill 不要求每个项目按同一顺序运行。

常见起点包括：

- 只有研究兴趣：从 `rw-research-router` 开始。
- 已有 PDF：从 `rw-paper-extractor` 开始。
- 已有方案：从 `rw-research-referee` 开始。
- 已有草稿：从 `rw-phd-write` 开始。
- 不知道当前问题属于哪里：回到 `rw-research-router`。

一次只处理当前瓶颈。完成后再决定下一步。

## 19. 5种常见推进路径

### 路径A：从研究兴趣开始

```text
研究兴趣
→ Router 界定问题
→ 文献发现和检索策略
→ Paper Extractor 建立证据
→ Referee 检查设计
→ PhD Write 组织写作
```

### 路径B：从一批论文开始

```text
PDF 和补充材料
→ Paper Extractor 提取字段和定位
→ 建立证据关系
→ 核验关键主张
→ Router 判断下一步
```

### 路径C：从研究方案开始

```text
研究问题和方案
→ Referee 检查偏倚和替代解释
→ 修改设计
→ 再次审查
→ 进入执行或写作
```

### 路径D：从论文草稿开始

```text
草稿、来源和修改要求
→ PhD Write 判断章节功能
→ 核对主张和来源
→ 在批准范围内修改
→ 引用核验
→ 交付检查
```

### 路径E：从审稿意见开始

```text
审稿意见和原稿
→ PhD Write 拆分意见
→ 确认修改范围
→ 修改指定位置
→ 核验主张和引用
→ 准备回复和投稿材料
```

## 20. 建议的项目材料结构

下面是整理建议，Skill 运行不依赖这种目录结构。

```text
research-project/
├── 00-original-question/
├── 01-protocol-and-design/
├── 02-papers/
├── 03-extraction/
├── 04-evidence/
├── 05-analysis/
├── 06-writing/
├── 07-review/
└── 08-delivery/
```

保留原始文件，不在唯一副本上直接改写。Agent 生成的提取、审查和写作文件放到对应目录。

## 21. 跨对话继续

聊天上下文不等于研究项目档案。关闭对话、切换 Agent 或隔一段时间继续时，不能只依赖聊天记录。

可以让 Router 保存项目状态：

```text
$rw-research-router
请保存当前研究项目的接续状态。
记录原问题、已有材料、已确认事实、当前判断、未知事项、
已否定方向、当前阶段、停止条件和下一步。
不要把未确认推断写成事实。
```

内部会使用 `rw-research-passport` 的项目状态结构。新对话中，把状态文件和当前材料一起提供：

```text
$rw-research-router
这是上次保存的项目状态和本轮新增材料。
先检查上游材料是否变化，再告诉我当前应继续哪一步。
```

## 22. 上游变化和 STALE

如果原始 PDF、研究问题、提取配置、数据、草稿或修改范围发生变化，旧结果可能不再有效。

RW Research Skill 使用 hash 和 `STALE` 状态表达这类变化：

- 来源变了，旧提取可能失效。
- 提取变了，旧证据图和主张核验可能失效。
- 草稿变了，旧引用、风格和交付审计可能失效。
- 研究问题变了，旧检索式和设计判断可能不再适用。

不要把旧结果直接复制到新版本。先说明变化，再重跑受影响的步骤。

## 23. 一次任务如何交接

每次任务结束时，要求 Agent 给出：

- 当前任务。
- 使用的材料和版本。
- 已确认事实。
- 当前判断。
- 未知事项。
- 没有处理的内容。
- 停止条件。
- 一个下一步。

可复制提示：

```text
请把本轮结果整理成可交接状态。
区分已确认事实、当前判断和未知事项。
列出使用的文件、没有处理的内容、停止条件和一个下一步。
```

# 80—100：建立证据和验收

## 24. 不把“有引用”当作“主张成立”

引用检查至少有3层：

1. 来源是否真实存在，身份是否正确。
2. 引用格式是否符合当前要求。
3. 来源是否支持句子中的具体主张。

一篇论文存在，不代表它支持你写出的全部句子。来源支持范围小于句子范围时，应收窄句子。

可复制提示：

```text
$rw-paper-extractor
请核验下面3条主张。
对每条主张检查来源身份、原文位置和支持范围，
并给出 PASS、REVIEW 或 BLOCK。
无法定位原文的关键主张不能标记为已核验。
```

## 25. PDF 精读的验收

`rw-paper-extractor` 可以建立 Paper Case，包括来源 hash、配置 hash、文本位置、章节候选、图表候选和阶段状态。

精读结果应保留：

- 题名、作者、年份、来源和稳定标识。
- 使用的提取 schema 版本。
- 研究问题、设计、样本、变量、方法、结果和限制。
- 页码、段落、图、表或补充材料位置。
- 自动识别的置信度和复核状态。
- 未报告、不适用、无法判断和缺失材料。

跨页表格的拼接图是阅读产物，不替代原始页码和位置。

## 26. 研究审查的验收

`rw-research-referee` 的结果不应只是“问题清单”。每个问题需要包含：

- 问题是什么。
- 证据或材料位置在哪里。
- 它怎样影响最强结论。
- 可以怎样修复。
- 哪些部分无法修复。
- 需要什么新证据才能排除替代解释。

语言粗糙不等于方法错误。报告缺失也不自动证明研究没有实施，但会限制审查。

## 27. 写作的验收

`rw-phd-write` 的结果应满足：

- 当前章节或段落的功能得到说明。
- 论点、证据、作者解释、重要性和研究问题连接能被区分。
- Methods、Results 和 Discussion 没有混写。
- 事实性主张有用户材料或可核验来源。
- 没有证据时，不生成引用或 DOI。
- 没有为了流畅删除限制、反例和不确定性。
- 修改范围和未修改内容得到说明。

最终提交前，作者仍需确认文字、引用、作者贡献、利益冲突和 AI 使用声明。

## 28. 什么时候算到达100

你能完成下面的动作，就达到了这份手册所说的“100”：

- [ ] 能独立安装和更新4个公开入口。
- [ ] 不知道从哪里开始时会使用 Router。
- [ ] 能给出当前目标、已有材料和不能越过的范围。
- [ ] 能用 Paper Extractor 提取带原文位置的证据。
- [ ] 能用 Referee 检查设计和结论边界。
- [ ] 能用 PhD Write 在来源和范围内写作。
- [ ] 能保存项目状态并在新对话中继续。
- [ ] 知道上游变化会让旧结果变成 STALE。
- [ ] 能区分来源存在、引用格式和主张支持。
- [ ] 在证据不足、专业审批或访问受限时会停止。

达到这里所说的“100”，意味着你能控制输入、范围、证据、版本和下一步。研究责任仍由你承担。

# 常见问题

## 29. `node` 或 `npx` 找不到

先安装 Node.js LTS：

[Node.js 下载页](https://nodejs.org/en/download)

安装后关闭并重新打开 Terminal 或 PowerShell，再运行：

```bash
node --version
npx --version
```

## 30. 安装后没有看到 Skill

按顺序检查：

1. 新建一个对话。
2. 再次运行安装命令。
3. 确认仓库名称是 `rolandwonglonam/rw-research-skill`。
4. 在 Codex 中使用 `$rw-research-router`。
5. 在不支持 `$` 语法的 Agent 中，直接要求使用该 Skill。

## 31. 为什么只看到4个入口

这是 v0.11.0 的默认行为。普通用户只使用4个公开入口。17个内部 Skill 仍在安装目录中，由公开入口接续，也保留旧名称兼容。

不要为了“完整”而主动显示和学习全部内部 Skill。

## 32. Agent 要我自己选择内部 Skill

输入：

```text
请回到 rw-research-router。
不要展示内部 Skill 清单。
根据我当前的瓶颈只选择一个主入口，并说明需要的材料和停止条件。
```

## 33. Agent 一次给了整条科研流程

输入：

```text
本轮只处理当前一步。
请说明我现在只需要做什么、怎样算完成，以及完成后的一个下一步。
```

## 34. Agent 编造了论文、DOI 或数据

停止使用该部分结果。要求重新执行：

```text
删除无法回到用户材料或可核验来源的事实性主张。
对保留的主张提供来源身份和原文位置。
无法核验的部分标记为未知，不要补造。
```

需要时使用主张核验流程。没有原文位置的关键字段不能标记为已核验。

## 35. PDF 只有摘要或无法访问全文

可以处理摘要实际报告的内容，但不能填写正文专属字段，也不能推断随机化、盲法、模型设定或缺失数据处理。

不要要求 Agent 绕过访问限制。

## 36. 网络不可用

Skill 可以继续使用包内的稳定方法和用户提供的材料。需要当前文献、API、期刊政策、费用或报告规范的部分，应标记为待核验。

网络恢复后，回到官方来源核验。

## 37. Agent 读不到文件

检查：

- 文件是否已经上传或放在 Agent 可以读取的位置。
- 是否提供了明确路径。
- 是否只有云端占位文件，没有下载到本机。
- 是否受到账户、系统或机构权限限制。
- 文件是否损坏或加密。

不要因为某个目录不可读，就假装已经读取其中内容。

## 38. 新对话忘记了上次内容

聊天记录不能代替项目档案。把上次的项目状态、关键产物和新增材料一起提供，再让 Router 判断当前步骤。

## 39. 输出太长

可以要求：

```text
先给当前任务、我现在只做的1件事和本轮完成标准。
证据边界、未知事项和停止条件不能省略。
其余细节放在后面。
```

如果需要，也可以明确要求“ADHD 友好输出”。该模式只在用户明确要求时启用。

## 40. 能不能直接让它写完整论文

可以处理论文写作任务，但前提是研究问题、材料、来源、章节功能和修改范围足够清楚。

没有来源时不能生成事实性引用。没有研究结果时不能写成研究已经完成。最终作者责任仍由研究者承担。

# 当前版本的验证边界

## 41. 已确认的发布事实

根据 v0.11.0 仓库和 2026-07-23 验证记录：

- 发布包包含21个科研 Skill。
- 默认公开入口为4个。
- License 为 Apache-2.0。
- 公开包标记为 preview。
- 21个 Skill 的静态自检通过。
- 默认安装列表和安装后的普通列表显示4个入口。
- 4个入口各运行了1个合成降级场景。
- 公共内容检查没有发现当前检查规则定义的隐私失败项。

这些结果说明当前发布结构、安装入口和部分确定性行为符合仓库记录。

## 42. 不能从验证记录推出什么

当前记录不能证明：

- 所有模型都会执行相同行为。
- 所有 PDF 版式都能自动提取。
- 所有研究问题都能得到正确路由。
- Skill 已经改善真实科研质量。
- 静态自检等于模型完成了全部行为合同。
- 合成任务结果等于真实学生、真实论文和真实数据上的效果。
- Agent 输出可以替代导师、统计师、伦理委员会或期刊确认。

v0.7.1 的跨模型记录只适用于当时保存的8个合成任务、模型入口和版本。v0.11.0 尚未运行完整跨模型矩阵，不能把旧结果直接写成当前整包效果。

## 43. 学生应怎样看待输出

把输出分成4类：

| 类型 | 含义 | 下一步 |
| --- | --- | --- |
| 已确认事实 | 能回到用户材料或已核验来源 | 可以继续使用，保留来源位置 |
| 当前判断 | 基于现有事实作出的解释 | 检查假设和替代解释 |
| 未知事项 | 当前材料不能回答 | 补材料或保持未知 |
| 停止条件 | 再继续会超出证据、权限或专业边界 | 停止并交给相应人员或来源 |

这4类没有分开时，不要把输出直接放进论文或研究方案。

# 术语表

## Agent

读取用户输入、工具结果和 Skill 说明，并执行任务的 AI 工作环境。具体能力和数据处理方式取决于所用产品和模型提供方。

## Agent Skill

一组供 Agent 读取的任务说明、方法、停止条件和资源。它依附于 Agent 运行。

## 入口

用户直接调用的 Skill。v0.11.0 默认提供4个公开入口，内部流程由入口接续。

## Router

判断当前研究阶段和瓶颈，并选择一个主任务的入口。

## Paper Case

围绕一篇 PDF 保存来源 hash、配置、文本位置、章节、图表、阶段产物和复核状态的工作区结构。

## Locator

能够回到原文的位置，例如页码、段落、表、图、补充材料、bbox 或 text unit。

## Hash

根据文件内容生成的标识。文件变化时，hash 通常也会变化，可用于判断旧结果是否仍对应当前版本。

## STALE

上游材料已经变化，下游结果需要重新检查的状态。

## PASS

当前证据足以通过指定门禁。它只适用于被检查的主张和版本，不表示整篇论文正确。

## REVIEW

存在部分支持、定位问题、范围不一致或其他需要人工复核的情况。

## BLOCK

当前证据不足、来源不支持、访问受限或存在阻断问题，不能继续把该结果当作已核验内容。

## 停止条件

继续任务会超出证据、权限、访问条件或专业责任边界时，必须停止的位置。
